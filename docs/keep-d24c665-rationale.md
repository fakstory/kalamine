# Why `d24c665` should be kept even after the xkbcommon fix

Commit `d24c665` ("fix: xkalamine install/remove on Wayland") on
`fix/xkalamine-wayland` stops kalamine from writing
`~/.config/xkb/rules/evdev` in user-mode installs. After landing the
xkbcommon fix in `third_party/xkbcommon@fix/silent-include-failure`
(commits `6abfad34` + `cd8cc2a1`), the canonical user rules content
`! include %S/evdev` works correctly. The natural question is: should
we revert `d24c665` and let kalamine write the user rules file again?

**Answer: no. Keep `d24c665`.** Here is why.

## 1. xkbcommon fixes the symptom, not the underlying coupling

The xkbcommon patch corrects a misleading filename in diagnostics and
makes include failures observable at default verbosity. It does **not**
change the lookup precedence: a user `rules/evdev` still shadows the
system one. So the "user wins, system loses" hazard is intrinsic to the
include-path design — the only safe ways to deal with it are:

- not writing user rules at all (what `d24c665` does), or
- writing a *correct, complete* user rules file that delegates via
  `! include %S/evdev` (what kalamine used to do).

Both work; the first is strictly simpler.

## 2. Fewer moving parts at the user level

`d24c665` writes only `~/.config/xkb/rules/evdev.xml` (the index GUIs
read) and the user's symbols file. It does not write the text rules
file. That means:

- The user's home directory has fewer files generated by kalamine.
- There is no `! include` indirection that has to be kept in sync with
  whatever the system rules file expects.
- A future xkbcommon change to `%S` semantics, the include grammar, or
  the path stack cannot break kalamine: there is nothing in the user's
  rules to break.

Reverting `d24c665` would re-introduce a tight coupling between
kalamine's generated text and xkbcommon's rules-file grammar, for no
functional gain.

## 3. Compositor and distro variability

`DFLT_XKB_CONFIG_ROOT` is a compile-time default of libxkbcommon. On
this Arch system the vendored fork resolves it to
`/usr/share/xkeyboard-config-2`, while plain xkeyboard-config layouts
live under `/usr/share/X11/xkb`. Different distros, different
compositors (Mutter, KWin, niri, Sway), and different libxkbcommon
builds can all carry different defaults and different `.pre`/`.post`
expectations.

A user `rules/evdev` containing `! include %S/evdev` *assumes* `%S`
points to a directory that contains a working `evdev` rules file.
That assumption is exactly the kind of cross-stack coupling
`~/AGENTS.md` warns against ("never break legacy"). Not writing the
file at all sidesteps the entire question.

## 4. Mixed root + user installs stay safe

The bug originally chased was the root-then-user install sequence: a
user with a previously-installed system layout (sudo) re-installs in
user mode. With `d24c665`:

- System symbols + system rules are still in place (untouched by
  user-mode kalamine).
- User-mode kalamine adds the user symbols file and the `evdev.xml`
  index entry.
- xkbcommon walks user XKB path → user rules absent → system rules
  picked up unchanged.

That is exactly the desired fall-through. Re-introducing user-rules
write would re-create the precedence question every time the user did
the same dual install, even if the include now works.

## 5. Test coverage already follows `d24c665`

`tests/test_xkalamine.py` (added with `a5f7353`) exercises the
no-user-rules path. Reverting `d24c665` would invalidate those tests
and require a fresh test matrix covering the include-delegation path
and the per-distro `%S` resolution. That is a non-trivial cost for a
behavior change that produces no observable user benefit.

## 6. The xkbcommon fix is still worth keeping

Even though `d24c665` makes the bug unreachable from kalamine, the
xkbcommon diagnostic fix is independently valuable:

- Anyone running into the same partial-rules buffer reuse from a
  hand-written user rules file (no kalamine involved) gets a correct
  filename in errors.
- The added `log_warn`/`log_dbg` lines surface include failures at
  default verbosity, which used to be impossible without rebuilding.
- The fix is a one-line buffer split with no API/ABI impact, easily
  upstreamable.

So: ship the xkbcommon fix, **leave `d24c665` in place**, and let the
two coexist. Kalamine stays simple; xkbcommon stops lying about
filenames.

## TL;DR

`d24c665` is not a workaround that becomes obsolete once xkbcommon is
fixed. It is a deliberate choice to avoid generating a user rules
file at all — which removes a class of cross-stack coupling that the
xkbcommon fix cannot eliminate, only make less confusing. Keep it.
